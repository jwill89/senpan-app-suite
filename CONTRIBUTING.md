## Getting set up

**Prerequisites:** Node 24+ and Go (version per `src/go.mod`).

```bash
cd src && go run . -addr :8080 -db ../data/bingo.sqlite   # backend
cd frontend && npm install && npm run gen:types && npm run dev   # frontend
```

`frontend/src/types/api.generated.ts` is generated by tygo and **gitignored** â€”
run `npm run gen:types` after a fresh clone and whenever you change Go model
types, or `typecheck`/`test`/`build` will fail.

## Branches & merge requests

- The default branch is `main`. Don't commit directly to it â€” work on a feature
  branch and open a merge request.
- Keep each MR focused on one logical change. Rebase on `main` before requesting
  review so CI runs against current code.
- Commit messages: a short, capitalized, imperative subject line (e.g.
  `Add Garapon festival lottery-drum feature.`), with a body when the *why* isn't
  obvious from the diff.

## Before you push â€” run the same gates as CI

CI fails the pipeline on any of these, so run them locally first.

**Frontend (in `frontend/`):**

```bash
npm run lint:check     # ESLint (no --fix)
npm run typecheck      # vue-tsc
npm run test           # Vitest (one-shot)
npm run build          # production build
```

**Backend (in `src/`):**

```bash
golangci-lint run ./...
go build ./...
go vet ./...
go test ./...
```

(CI also runs `gen:types` and `govulncheck`; see [`.gitlab-ci.yml`](.gitlab-ci.yml)
and [`.github/workflows/ci.yml`](.github/workflows/ci.yml).)

> **One command:** maintainers can run `scripts/check.ps1` (PowerShell) to run
> every gate above â€” backend `golangci-lint`/build/vet/test, then frontend
> `gen:types`/lint/typecheck/test/build â€” and stop at the first failure.
> `golangci-lint` is **not** covered by `go build`/`go vet`, so don't skip it.
> (`scripts/` is gitignored local tooling â€” install golangci-lint with
> `go install github.com/golangci/golangci-lint/v2/cmd/golangci-lint@v2.12.2`.)

## Conventions

- **Match the surrounding code.** Mirror the existing naming, comment density,
  and idioms of the file you're editing rather than introducing a new style.
- **Tests live next to the code.** Frontend: colocated `src/**/*.test.ts`
  (Vitest + Vue Test Utils, jsdom). Backend: `_test.go` in the same package. Add
  or update tests with every behaviour change. See the **Testing** section of
  [`AGENTS.md`](AGENTS.md) for conventions.
- **Types flow from Go.** Backend model changes must be followed by
  `npm run gen:types`; don't hand-edit `api.generated.ts`.
- **Theming is structured tokens**, not free-form CSS â€” edit the token list in
  the three places that must stay in sync (`frontend/src/assets/styles/tokens.css`,
  `store.themeTokenOrder` in Go, `THEME_TOKEN_GROUPS` in `lib/theme-tokens.ts`).

## Versioning & changelog

The frontend and backend are versioned **independently** with semver
(MAJOR = breaking API change, MINOR = backward-compatible addition, PATCH = fix).
When your change ships:

1. Bump the matching version source â€” `frontend/package.json` `"version"` and/or
   `src/internal/version/version.go` (`Version`).
2. Add an entry under the matching section of [`CHANGELOG.md`](CHANGELOG.md)
   (newest first; _Added / Changed / Fixed / Removed_).

The admin dashboard shows both versions and flags a major-version mismatch, so
keeping them in step matters.

## Deploying

Deployment is maintainer-only via `scripts/deploy.ps1`
(`-Target frontend|backend|both`); see [`deploy/README.md`](deploy/README.md).
Database changes on production (outside the app's own migrations) are applied
manually and require explicit sign-off â€” never include them in a routine deploy.
